Lesson Introduction

Introduction to PostgreSQL

In this lesson, we move from application setup into the database layer and start working with PostgreSQL, one of the most widely used relational database systems in real-world web development. You already know SQL from SQL Server, so the big picture should feel familiar: tables hold data, rows represent records, columns define attributes, and SQL is still the language used to query and change information. What changes is mostly the platform and tooling, not the core relational ideas. PostgreSQL is especially popular in Flask, Django, and other modern backend stacks because it is reliable, powerful, standards-friendly, and well suited for both small projects and production systems. As we build RecipeShare, PostgreSQL will become the persistent storage behind our Flask API.

Mindset for today: you are not learning SQL from zero. You are learning how the SQL knowledge you already have transfers into a Python web application.

Why PostgreSQL?

PostgreSQL is an open-source relational database system that supports the same foundational ideas you used in SQL Server: tables, keys, joins, constraints, queries, indexes, and transactions. For a Flask application, it is a strong choice because it is stable, widely supported, and works well with Python tools such as SQLAlchemy and Flask-Migrate. It is also common in deployed applications, so it gives you experience with a production-style database instead of only a lightweight classroom database.

What feels familiar
  • CREATE TABLE, INSERT, SELECT, UPDATE, DELETE
  • Primary keys, foreign keys, and constraints
  • One-to-many and many-to-many relationships
  • Normalization and relational thinking
What is new in our stack
  • Using PostgreSQL instead of SQL Server
  • Connecting through a Flask application
  • Using SQLAlchemy models as a bridge to the database
  • Managing schema changes with migrations

Table

A table stores data about one kind of thing, such as users or recipes.

Row

A row is one record in that table, such as one user or one recipe.

Column

A column defines one attribute of that record, such as title, email, or prep time.

Working with pgAdmin on Windows

On your Windows VMs you can use pgAdmin as a graphical tool for working with PostgreSQL. pgAdmin lets you browse databases, inspect tables, view columns, run SQL, and verify that your schema matches what your Flask application expects. This is especially helpful early on, because you can see the database visually while you are also learning models and migrations in Python.

Typical connection values

  • Host: localhost
  • Port: 5432
  • Maintenance database: postgres or recipeshare
  • Username: your PostgreSQL username or postgres
  • Password: your PostgreSQL password or postgres

What you should do in pgAdmin

  • connect to the local PostgreSQL server
  • expand Databases and open recipeshare
  • inspect the users and recipes tables
  • right-click a table to view data or properties
  • open the Query Tool and run SQL manually
1

Open pgAdmin

Launch pgAdmin from the Windows Start menu and connect to your local PostgreSQL server.

2

Locate your database

Expand Servers → PostgreSQL → Databases and select recipeshare.

3

Inspect tables

Expand Schemas → public → Tables to see the tables created by your migration.

4

Run SQL

Use the Query Tool to experiment with SQL such as selecting recipes or joining recipes to users.

SELECT r.title, u.username
FROM recipes r
JOIN users u ON r.user_id = u.id;
Teaching value: pgAdmin gives you an easy visual bridge from SQL concepts you already know to the live PostgreSQL database that supports their Flask application.

Working from psql on Windows VMs

You should also learn how to work from the command line using psql, the PostgreSQL interactive terminal. This is useful when a GUI is unavailable, when you want to follow written instructions exactly, or when you need a quick way to test SQL commands. On the Windows VMs, you may need to add the PostgreSQL runtime folder to the command prompt PATH before psql can be launched directly.

Temporary PATH update in Command Prompt

In a regular Windows Command Prompt, you can run:

set PATH=%PATH%;C:\Program Files\PostgreSQL\17\pgAdmin 4\runtime

After that, you can try:

psql -U postgres
psql -U postgres -d recipeshare

Basic Database Operations

Even though our Flask app will often talk to PostgreSQL through SQLAlchemy, it is still important to understand the core SQL operations underneath. The application layer does not replace the database; it sits on top of it. When we define a model, create a migration, or save a new recipe, those actions eventually become SQL statements against the database.

1

Create a database

Before we create tables, we need a database to hold them.

CREATE DATABASE recipeshare;
2

Create a table

A table definition describes the structure of the data: column names, data types, keys, and rules such as NOT NULL or UNIQUE. 

CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(80) NOT NULL UNIQUE,
    email VARCHAR(120) NOT NULL UNIQUE
);
3

Create a related table

The recipes table connects each recipe to a user with a foreign key. This is the same relational idea you already know from SQL Server. 

CREATE TABLE recipes (
    id SERIAL PRIMARY KEY,
    title VARCHAR(150) NOT NULL,
    description TEXT NOT NULL,
    instructions TEXT NOT NULL,
    prep_time INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    CONSTRAINT fk_recipes_user
        FOREIGN KEY (user_id)
        REFERENCES users(id)
);
4

Insert rows

INSERT INTO users (username, email)
VALUES ('alice', 'alice@example.com');

INSERT INTO recipes (title, description, instructions, prep_time, user_id)
VALUES (
    'Pasta Primavera',
    'Fresh pasta with vegetables',
    'Boil pasta, saute vegetables, combine, and serve.',
    25,
    1
);
5

Read data

SELECT id, title, prep_time
FROM recipes;

SELECT r.title, u.username
FROM recipes r
JOIN users u ON r.user_id = u.id;
6

Update and delete

UPDATE recipes
SET prep_time = 30
WHERE id = 1;

DELETE FROM recipes
WHERE id = 1;

Illustration: How the Tables Connect

This is a one-to-many relationship: one user can own many recipes, but each recipe belongs to one user.

Common PostgreSQL Data Types

Type Use in our app
INTEGER Numeric values like prep time or IDs
SERIAL Auto-incrementing integer primary key
VARCHAR(n) Short text such as usernames or recipe titles
TEXT Longer text such as descriptions and instructions
TIMESTAMP Dates and times, such as when a recipe was created
BOOLEAN True/false values later, for flags and settings

Constraints You Should Remember

  • PRIMARY KEY uniquely identifies a row
  • FOREIGN KEY connects one table to another
  • NOT NULL requires a value
  • UNIQUE prevents duplicates, such as duplicate usernames
  • CHECK can enforce rules, such as positive prep time
CREATE TABLE recipes (
    id SERIAL PRIMARY KEY,
    title VARCHAR(150) NOT NULL,
    prep_time INTEGER CHECK (prep_time >= 0)
);

How This Fits into Flask

In plain SQL, we create tables and query them directly. In a Flask application, we usually define Python classes that represent those tables. SQLAlchemy then maps Python objects to relational tables. So a class like User becomes a users table, and a class like Recipe becomes a recipes table. When we add a new object in Python and commit the session, SQLAlchemy generates the corresponding SQL behind the scenes. 

class User(db.Model):
    __tablename__ = "users"

    id = db.Column(db.Integer, primary_key=True)
    username = db.Column(db.String(80), nullable=False, unique=True)
    email = db.Column(db.String(120), nullable=False, unique=True)

That does not mean SQL disappears. It means our application gets a higher-level, Python-friendly way to work with relational data. The database still matters, and understanding the underlying SQL will make SQLAlchemy much easier to reason about.

Migrations: Creating and Changing Tables Safely

In classroom demos, it is tempting to drop tables and recreate everything from scratch. In real applications, we want a safer process. That is why we use migrations. A migration records how the database schema changes over time. If we add a new column, rename a table, or create a new relationship, the migration system generates a versioned change that can be applied consistently. 

flask db init
flask db migrate -m "Create users and recipes tables"
flask db upgrade
Key idea: changing your Python model class does not automatically change the database. A migration is the bridge between your updated code and the actual schema in PostgreSQL.
PostgreSQL + Flask + SQLAlchemy gives us a clean bridge from relational database theory to real web application development.
