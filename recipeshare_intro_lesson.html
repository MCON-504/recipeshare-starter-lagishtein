Why PostgreSQL?

PostgreSQL is a powerful relational database that is widely used in professional web applications because it is reliable, fast, and designed for real multi-user systems. While SQLite is excellent for quick demos and very small projects, PostgreSQL gives us a more realistic environment for the kind of applications we would actually deploy. It supports strong relational modeling, solid data integrity rules, indexing, and concurrent access by many users. It also works especially well with Flask when paired with SQLAlchemy, because our Python models can map cleanly to database tables. Using PostgreSQL now helps you practice the same workflow used in many production applications: configure the database connection, define models, create migrations, and query real persisted data. In other words, this lesson is where our Flask app stops being a toy and starts behaving like an actual backend system.

The core data model

We are keeping the first version intentionally small. That helps us focus on the workflow of building the backend instead of drowning in too many tables. For now, we only need two models: User and Recipe. A user can have many recipes, and each recipe belongs to exactly one user.

User

  • id — primary key
  • username — unique display name
  • email — unique email address

In the next lesson, this model will also gain password-related fields for authentication.

Recipe

  • id — primary key
  • title — recipe title
  • description — short summary
  • instructions — preparation steps
  • prep_time — number of minutes
  • created_at — creation timestamp
  • user_id — foreign key to the author
Relationship Meaning in plain language Why it matters
One user → many recipes One person can create multiple recipe entries. This mirrors real application ownership and prepares us for auth.
Each recipe → one user Every recipe has a single author in the database. This gives us a clean foreign key and a natural query pattern.

First endpoints

Because we are preparing for a React front end, our first routes should return JSON instead of rendered HTML pages. That makes the backend reusable and keeps responsibilities separate: Flask provides data and business logic, while React will later provide the user interface.

Where migrations fit in

A very common beginner mistake is to assume that changing a model class automatically changes the database. It does not. The model describes what the schema should look like, but migrations are the mechanism that records and applies those schema changes. That means our workflow becomes: change the model, generate a migration, and upgrade the database. This is how teams safely evolve database-backed applications over time.

flask db init flask db migrate -m "Create user and recipe tables" flask db upgrade
Key idea: the Python code and the database schema are related, but they are not the same thing. Migrations are the bridge between them.

Setting up a Run/Debug Configuration in IntelliJ

To make development smoother, create a dedicated Run/Debug configuration for the Flask app in IntelliJ. This lets you start the server with one click, reuse the same environment variables every time, and place breakpoints in your routes and model code for debugging. Since our application is started through run.py, IntelliJ only needs to know which interpreter to use, which file exposes the Flask app, and which environment variables should be available when the server starts.

Recommended values

  • Name: RecipeShare Flask
  • Interpreter: your project virtual environment
  • Target / Script / FLASK_APP: run.py
  • Working directory: project root folder
  • Host: 127.0.0.1
  • Port: 5000

Environment variables

A typical local setup might include:

FLASK_APP=run.py FLASK_DEBUG=1 DATABASE_URL=postgresql://localuser@localhost/recipeshare

If your project loads values from .env, IntelliJ can still use them as long as your app reads that file when starting.

Setting up a Database View in IntelliJ

IntelliJ’s Database tool window gives you a live view of your PostgreSQL database. This is extremely useful in a lesson like this because students can connect the abstract ideas of models and migrations to something concrete: real tables, real rows, and real queries. After creating the connection, you can browse schemas, inspect table structures, run SQL manually, and verify that your Flask application is actually saving data where you expect it to.

Connection values for this project

  • Data Source: PostgreSQL
  • Host: 127.0.0.1 or localhost
  • Port: 5432
  • Database: recipeshare
  • User: localuser (or your local PostgreSQL user)
  • Password: use your PostgreSQL password if required

What students should look for

  • the users table created by the migration
  • the recipes table and its columns
  • the foreign key from recipes.user_id to users.id
  • seeded rows after running seed.py
  • changes after inserting data through the API
